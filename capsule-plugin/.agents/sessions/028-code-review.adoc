== Architecture — Points forts

=== ✅ Interfaces + NoOp pattern
Chaque couche critique (TtsEngine, PlaywrightCapture, ManimEngine,
ManimVideoMixer, ManimSlideReplacer, ManimParallelRenderer) suit le
pattern **interface + Impl réelle + NoOp fallback**.
Cela permet :
- Tests sans dépendances externes (playwright, ffmpeg, manim, piper, espeak)
- Graceful degradation en production si un binaire manque
- Injection facile dans les tests

=== ✅ Config multi-source (CAP-13)
Mécanisme à 4 niveaux bien conçu : ENV < gradle.properties < YAML < CLI.
Le `CapsuleConfigMerger` gère proprement le cas avec/sans fichier YAML.
Pattern aligné sur plantuml-gradle — cohérence cross-borough.

=== ✅ Taxonomie EPIC K
8 tâches Gradle avec verbes standardisés :
- `generateCapsuleScript`, `generateCapsule`, `generateCapsuleVideo`
- `deployCapsule`
- `collectCapsuleContext`, `transformCapsuleContext`
- `scaffoldCapsuleContext`, `collectCapsuleRetrieve`

=== ✅ Tests solides
- 173 unit tests (JUnit5) — couverture des cas nominaux + fallbacks
- 8 functional tests (GradleTestKit) — vérification des tâches
- 27+ scénarios Cucumber BDD — @config, @manim, @integration, @manim @parallel
- Tests de contrainte : vérification signature WebM, taille MP3, etc.

=== ✅ Parallélisation Manim
`ManimParallelRendererImpl` avec thread pool pour le rendu simultané
des slides MANIM multiples. Fallback séquentiel via `NoOpManimParallelRenderer`.

=== ✅ Gestion des placeholders
Tous les moteurs NoOp produisent des fichiers textes identifiables
(`# TTS PLACEHOLDER`, `# MANIM PLACEHOLDER`, `# PLAYWRIGHT CAPTURE PLACEHOLDER`).
La méthode `isPlaceholder()` dans CapsuleVideoTask permet de les détecter.
Le `isValidWebM()` dans CapsuleDistribTask vérifie la signature EBML
avant de crop, évitant de passer des placeholders à ffmpeg.

== Problèmes identifiés

=== 🔴 DRY — ffprobeDuration dupliqué 3 fois

Même méthode `ffprobeDuration()` dans :
1. `CapsuleVideoTask` (l.708-717) — private
2. `ManimEngineImpl` (l.65-79) — override
3. `ManimVideoMixerImpl` (l.106-120) — override

*Solution* : Extraire dans un `object FfprobeUtils` ou `FFmpegProbe` utilitaire.

=== 🔴 DRY — extractTopLevelSections dupliqué 2 fois

Logique identique de parsing HTML section dans :
1. `CapsuleVideoTask` (l.129-164) — companion
2. `ManimSlideReplacer` (l.113-145) — private

*Solution* : Factoriser dans une classe utilitaire partagée (ex: `HtmlSectionExtractor`).

=== 🔴 DRY — wavToMp3 dupliqué 2 fois

Même méthode de conversion WAV→MP3 dans :
1. `PiperTtsEngine` (l.62-81) — private
2. `EspeakTtsEngine` (l.147-164) — private

*Solution* : Extraire dans une fonction utilitaire ou un helper object.

=== 🟡 Magic numbers — parallélisme en dur

- `captureSlideParallel()` dans CapsuleVideoTask (l.523) : `Executors.newFixedThreadPool(4)`
- `resolveManimParallelRenderer()` dans CapsuleManager (l.139) : `parallelism = 4`

Le 4 est un bon default mais devrait être configurable via l'extension.

=== 🟡 Code mort — isPlaceholder()

La méthode `isPlaceholder()` (l.616-618) est définie mais **n'est jamais appelée**.
Probablement un vestige d'un refactoring.

=== 🟡 println au lieu de logger

`PlaywrightCaptureImpl` (l.75) :
```kotlin
println("  Playwright: hasAudio=${hasAudioElements} audioCount=${audioIds.size} slides=$slideCount")
```
Devrait être `logger.lifecycle(...)`.

=== 🟡 Resource leak potentiel — Playwright

Dans `CapsuleVideoTask.execute()` :
- `deckCapture.capture()` crée playwright/browser/context
- `deckCapture.close()` est appelé **dans le try** (l.494)
- Si une exception survient entre capture et close (ex: ligne 496-504),
  le close() n'est pas garanti → fuite de processus Chromium

Dans `captureSlideParallel()` (l.514-561) :
- `capture.close()` est dans un `finally` (correct ✓) mais capture est créée
  dans le submit() — chaque thread ouvre/ferme sa propre instance.

*Solution* : Wrap dans `use {}` ou garantir close() dans un finally block.

=== 🟡 Resource leak — concat-{timestamp}.txt

Dans `concatWebmFiles()` (l.61-76) :
- Le fichier `concat-{timestamp}.txt` est créé puis `.delete()` après usage
- Si ffmpeg plante entre les deux, le fichier temporaire reste

*Solution* : Utiliser `File.deleteOnExit()` ou un bloc try/finally.

=== 🟡 lateinit var sans garde

Toutes les tâches utilisent `lateinit var capsuleExtension: CapsuleExtension`.
Si la tâche est exécutée sans avoir été configurée par le plugin → crash sec.

*Solution* : Optionnel : déléguer à `project.extensions.getByType()` avec lazy.

=== 🟡 ManimConfig.validate() — logique perfectible

```kotlin
if (quality.isNotBlank() && quality != "noop" && quality !in VALID_QUALITIES) { ... }
// ...
if (quality == "noop") { /* noop */ }
else if (quality.isBlank()) { errors.add(...) }
```

La logique est correcte mais difficile à lire. Suggestion : simplifier avec
`when(quality) { "noop" -> /* ok */ in VALID_QUALITIES -> /* ok */ else -> errors }`.

=== 🟢 Minime — `e: Exception` silencieux

Plusieurs catch blocks attrapent `Exception` sans log :
- `CapsulePlugin.kt` l.12 : `catch (_: Exception)` → juste debug log
- `FfprobeDuration` dans les 3 classes → retourne 0.0 silencieusement

C'est acceptable pour des soft-fallbacks mais mérite un `logger.debug()`.

=== 🟢 Minime — slider en compileOnly jamais référencé

build.gradle.kts : `compileOnly(libs.slider)` mais aucune import de slider
dans le code Kotlin. La dépendance est utilisée uniquement via
`project.plugins.apply("education.cccp.slider")` — réflexion.

C'est correct (le contrat est le deck HTML, pas l'API Kotlin),
mais la dépendance pourrait être `implementation` pour éviter les surprises.

=== 🟢 Minime — JDK version hardcodée

```kotlin
kotlin.jvmToolchain(JavaVersion.VERSION_24.ordinal)
```
24 est la version de dev. Devrait utiliser le toolchain du workspace BOM.

== Conclusion

pourrait être renforcée.
